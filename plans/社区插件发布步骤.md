# 社区插件发布步骤

## 目的

这份文档只保留“真正能成功发布”的路径，不记录中途走偏的尝试。文档已做脱敏处理，不包含账号名、邮箱、个人绝对路径或特定仓库所有者信息。

## 适用范围

适用于当前插件通过 `community.obsidian.md` 提交到社区插件目录的流程。

## 发布前硬性检查

在开始提交前，先确认以下条件全部满足：

1. 默认分支中的 `manifest.json` 已是目标发布版本。
2. 版本号在这些位置保持一致：
   - `manifest.json`
   - `package.json`
   - `versions.json`
3. GitHub Release 的 tag 使用精确版本号，不加 `v` 前缀。
4. GitHub Release 中至少包含三个单独资产：
   - `main.js`
   - `manifest.json`
   - `styles.css`
5. Release 里的 `manifest.json` 与仓库默认分支中的 `manifest.json` 一致。
6. `manifest.json` 的 `description` 不要包含 `Obsidian` 这个词。
7. README 中已写清插件用途、安装方式和主要功能。

## 成功发布的实际路径

### 第一步：先完成 GitHub Release

1. 在本地完成构建。
2. 确认版本号一致。
3. 推送默认分支。
4. 创建对应 tag。
5. 创建 GitHub Release。
6. 上传 `main.js`、`manifest.json`、`styles.css` 三个资产。

### 第二步：连接社区站与 GitHub

1. 打开 `https://community.obsidian.md/account/plugins`。
2. 连接 GitHub 账号。
3. 确认社区站已能识别当前仓库。

### 第三步：创建插件草稿

1. 点击 `New plugin`。
2. 输入插件仓库地址：

```text
<你的插件仓库 URL>
```

3. 勾选开发者政策确认项。
4. 勾选维护承诺确认项。
5. 点击提交，生成草稿条目。

### 第四步：让社区站抓取正确的 Release

如果社区站抓到的是旧版本，或者 review 中显示的不是当前 release：

1. 打开条目页面右上角菜单。
2. 点击 `Check for new releases`。
3. 反复确认最新 review 记录里：
   - 版本号是当前版本
   - commit 是当前默认分支对应提交

## 审查结果怎么看

### 必须处理的项目

只要出现 `Error`，就不要发布。

典型阻塞项包括：

- 描述文本不符合规则
- Release manifest 与仓库 manifest 不一致
- 缺少必需发布资产
- 自动审查发现源码层面的阻塞问题

### 可以带着发布的项目

以下通常不是阻塞项：

- `Recommendation`
- `Warning`
- 权限说明类提示，例如库枚举、剪贴板访问
- Release notes 缺失
- 构建证明缺失

这些可以后续再优化，不影响首次发布成功。

## 何时点击 Publish

只有当最新一条 review 满足以下条件时再点 `Publish`：

1. 最新 review 对应的是当前版本。
2. 最新 review 对应的是当前提交。
3. 状态不再是失败。
4. 没有阻塞发布的 `Error`。

点击 `Publish` 后，若页面从草稿状态切换为公开状态，通常会出现：

- `Publish` 按钮消失
- `View listing` 按钮出现
- 最新 review 状态显示完成

这说明发布已经成功。

## 发布后验证

发布完成后，按这个顺序验证：

1. 点击 `View listing`，确认公开页面可访问。
2. 在社区站条目页确认最新 review 已完成。
3. 在宿主应用的社区插件浏览器中搜索插件名。

注意：客户端搜索结果可能存在同步延迟。网页条目已公开，不代表客户端立刻能搜到。

## 不建议作为常规路径使用的按钮

### `Request review`

这个按钮更适合：

- 自动审查结果存在误报
- 需要人工介入解释某项结论

它不是正常首次发布的主要步骤。

## 发布失败时的最短修复路径

如果发布前的最新 review 失败，按这条路径处理：

1. 只看最新一条 review。
2. 修复其中的 `Error`。
3. 发布一个新版本。
4. 上传新的 Release 资产。
5. 回到社区站点击 `Check for new releases`。
6. 只等待最新版本通过，不再纠结旧版本失败记录。

## 当前项目后续维护建议

1. 首发成功后，不必为了清空所有 warning 立刻重构代码。
2. 先保证功能稳定、文档清楚、版本一致。
3. 后续再逐步处理：
   - release notes
   - 类型告警
   - 多窗口兼容性告警
   - CSS 细节告警
