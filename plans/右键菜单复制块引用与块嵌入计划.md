# 右键菜单复制块引用与块嵌入计划

## 目的

这份文档记录“给编辑器右键菜单新增复制块引用与复制块嵌入入口”的最终实施方案，供后续新的聊天窗口或新的工程代理直接续接。

## 需求背景

当前插件已经有：

1. `Copy current block reference`
2. `Copy current block embed`
3. 已有 UUID 复用逻辑
4. 无 UUID 时自动补 `id:: uuid` 的逻辑

当前还缺：

1. 在编辑器右键菜单中快速触发这两个复制动作的入口

## 已确认的产品决策

1. 右键入口只在编辑器中提供
2. 第一版不扩展到 Reading Mode、文件树或其它上下文菜单
3. 右键入口只对当前大纲源块生效
4. 不新增第二套 UUID 生成逻辑
5. 右键菜单必须复用现有命令的复制逻辑
6. 已有 `id:: uuid` 时绝不重新生成
7. 没有 `id:: uuid` 时继续按现有规则插入
8. 第一版不依赖鼠标右键坐标反推 Markdown 行
9. 第一版以当前编辑器光标所在行作为目标块

## 菜单项设计

新增两个右键菜单项：

1. `Copy block reference`
2. `Copy block embed`

命令面板中的已有命令名称保持不变：

1. `Copy current block reference`
2. `Copy current block embed`

这样区分更清楚：

1. 命令面板强调 `current`
2. 右键菜单天然就是当前块语境，不需要再重复写 `current`

## 适用范围

右键菜单项只在以下情况出现：

1. 当前处于 Markdown 编辑器上下文
2. 当前光标所在行符合源块规则
3. 当前行是以 `- ` 开头的无序列表块源行

不在以下情况出现：

1. 普通段落
2. 标题
3. 空行
4. 代码块
5. 文件树右键菜单
6. Reading Mode 页面普通右键

## 技术实现

使用 Obsidian 公开 API：

```ts
this.registerEvent(this.app.workspace.on('editor-menu', ...))
```

这是公开的编辑器右键菜单入口，不需要碰内部 Search 或 CodeMirror 私有菜单实现。

### 菜单注册规则

1. 在插件 `onload()` 中注册 `editor-menu`
2. 回调参数使用官方提供的：
   - `menu`
   - `editor`
   - `info`
3. 只做轻量判断，不读文件、不扫全库

### 当前块判断

通过当前光标所在行做判断：

1. 读取 `editor.getCursor().line`
2. 读取 `editor.getLine(line)`
3. 用当前插件已有源块规则判断是否匹配无序列表源行

匹配规则保持和当前命令逻辑一致：

```ts
/^\s*-\s(.+)/
```

### 为什么不做鼠标坐标定位

第一版不根据右键鼠标位置反推行号，原因是：

1. `editor-menu` 公开回调不提供右键事件坐标
2. 为了拿坐标去解析 CodeMirror DOM，会开始依赖更脆弱的内部结构
3. 当前功能重点是稳定增加快捷入口，不是做复杂右键定位系统

因此，第一版的稳定方案是：

1. 用户先把光标放在目标大纲块上
2. 再点右键使用菜单项

## 复制行为

右键菜单点击后，必须直接复用当前已有复制逻辑。

### `Copy block reference`

复制：

```md
((uuid))
```

### `Copy block embed`

复制：

```md
{{embed ((uuid))}}
```

### UUID 规则

1. 如果当前块已经有 `id:: uuid`，必须复用
2. 如果当前块还没有 `id:: uuid`，按当前既有规则生成并插入
3. 如果索引尚未更新，但编辑器文本中已存在 `id:: uuid`，也必须复用

## 代码结构要求

### 复制逻辑

现有：

1. `handleCopyBlockReference`
2. `handleCopyBlockEmbed`
3. `copyCurrentBlockSyntax`
4. `getOrCreateCurrentBlockId`

本轮要求：

1. 保留现有结构
2. 右键菜单点击后调用 `copyCurrentBlockSyntax`
3. 不单独复制一份“右键专用复制逻辑”

### 上下文类型

`editor-menu` 的第三个参数来自公开类型：

```ts
MarkdownView | MarkdownFileInfo
```

因此复制函数需要兼容这两种上下文。

核心只依赖：

1. `info.file`
2. `editor`

不需要引入额外复杂状态。

## 性能要求

这个功能必须保持非常轻量。

右键菜单打开时只允许：

1. 读取当前光标位置
2. 读取当前行文本
3. 进行一次正则判断
4. 向菜单中添加最多两个菜单项

右键菜单打开时不允许：

1. 扫描整个文件
2. 读取整个 vault
3. 查全量索引统计
4. 打开弹窗
5. 读取缓存文件
6. 进行任何渲染刷新

真正生成或复用 UUID 只在用户点击菜单项之后发生。

## 验证清单

### 功能验证

1. 光标在标准无序列表源块上，右键能看到 `Copy block reference`
2. 光标在标准无序列表源块上，右键能看到 `Copy block embed`
3. 光标在普通段落、标题、空行上，右键不显示这两个菜单项
4. 点击 `Copy block reference` 后，剪贴板内容为 `((uuid))`
5. 点击 `Copy block embed` 后，剪贴板内容为 `{{embed ((uuid))}}`
6. 已有 `id:: uuid` 的块通过右键复制时，不生成新 UUID
7. 没有 `id:: uuid` 的块通过右键复制时，会按现有规则补上 ID

### 回归验证

1. 命令面板中的 `Copy current block reference` 行为不变
2. 命令面板中的 `Copy current block embed` 行为不变
3. 现有大库索引、徽标、Back 按钮、渲染功能不受影响

### 工程验证

1. `npm run build` 通过
2. 没有新增明显性能卡顿

## 非目标

本轮不做以下内容：

1. 不支持鼠标右键坐标精确定位块
2. 不支持 Reading Mode 右键复制
3. 不支持文件树右键复制
4. 不新增设置项
5. 不改动源块识别规则
6. 不修改搜索功能
