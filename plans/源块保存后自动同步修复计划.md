# 源块保存后自动同步修复计划

## 目标

修复“源块内容修改并保存后，块引用 `((uuid))` 与块嵌入 `{{embed ((uuid))}}` 仍显示旧内容”的问题。

本次修复只支持“保存后自动同步”，不做“用户一边打字，其他页面实时联动刷新”。

## 核心规则

1. 同一个 UUID 如果有多个活动源块，只认“最新被修改的源块”为当前有效源块。
2. 所有块引用、块嵌入、自动补全、Back 跳转、源块引用计数徽标，统一以这个有效源块为显示基础。
3. 不自动改写旧备份页或其他重复源块的 Markdown 原文，只统一插件渲染结果与索引结果。
4. 自动更新以 Obsidian 的文件保存和文件变更事件为边界，不做编辑中的实时联动。

## 问题判断

当前问题主要在索引层，不是单纯的渲染层问题。

已确认的风险点：

- `BlockCache` 当前没有记录“源块所属文件最后修改时间”，导致同 UUID 多源块时，canonical 源块可能长期停留在旧备份页。
- `IndexService.parseFile()` 当前使用 `vault.cachedRead(file)`，在保存瞬间有机会读到旧缓存内容。
- 当前 canonical 选择逻辑会优先沿用之前的 canonical 源块，再退回路径排序，这不符合“最新修改源块优先”的规则。

## 详细实施方案

### 1. 索引数据结构扩展

在 `src/types.ts` 的 `BlockCache` 中新增：

- `sourceUpdatedAt: number`

含义固定为：

- 该源块所属 Markdown 文件在本次被索引解析时的 `TFile.stat.mtime`

要求：

- 所有 active 源块都必须带这个字段。
- stale 块、confirmed_deleted 块继续保留这个字段，用于历史回退与排序兼容。

### 2. 文件解析改为新鲜读取

在 `src/services/IndexService.ts` 中：

- `parseFile(file)` 从 `this.app.vault.cachedRead(file)` 改为 `this.app.vault.read(file)`

目标：

- 确保 `modify` 事件触发后，索引读取的是最新保存内容，而不是旧缓存。

边界：

- 这里只改索引解析读取方式。
- 其他只读预览缓存逻辑，例如引用预览行缓存，暂不改动。

### 3. 所有入索引的源块都记录 sourceUpdatedAt

在 `populateStateFromParsedFile()` 中写入 `normalizedBlock` 时：

- 增加 `sourceUpdatedAt: file.stat.mtime`

覆盖场景：

- 首次全量重建
- 启动后的 reconcile
- 单文件 `create`
- 单文件 `modify`
- `rename` 后重新解析

### 4. 手动创建源块时也补齐时间字段

在 `src/main.ts` 的 `getOrCreateCurrentBlockId()` 中，调用 `indexService.addBlock()` 时：

- 新增 `sourceUpdatedAt`
- 取值使用当前文件的 `file.stat.mtime`

目的：

- 保证通过命令或右键创建 `id:: uuid` 的场景也符合新索引结构。

### 5. canonical 源块选择规则重写

在 `src/services/IndexService.ts` 中重写 `pickCanonicalSourceBlock()`：

新的固定规则：

1. 按 `sourceUpdatedAt` 降序，最新修改优先。
2. 如果 `sourceUpdatedAt` 相同，按 `filePath` 升序。
3. 如果路径相同，按 `startLine` 升序。
4. 如果还相同，按 `endLine` 升序。

要求：

- 移除“优先沿用 previousBlock”的逻辑。
- 不能再因为之前已经选中过某个旧源块，就继续把它当成 canonical。

### 6. 活动源块列表顺序同步更新

`getActiveSourceBlocks(uuid)` 的返回顺序也要改成与 canonical 规则一致：

- 最新修改优先
- 再按路径与行号稳定排序

这样 `Back` 按钮出现多源块选择菜单时，用户看到的第一个位置就是当前实际有效源块。

### 7. 缓存兼容与迁移

当前缓存 schema 是 3。

本次实现优先采用兼容补齐方案：

- 读取旧缓存时，如果 `sourceUpdatedAt` 缺失：
  - 先尝试用 `FileIndexMeta.mtime`
  - 再退回 `lastSeenAt`
  - 最后兜底 `0`

如果实现中发现 schema 3 兼容处理不够清晰，再升级到 schema 4，并补一段从旧 schema 迁移的逻辑。

要求：

- 不要求用户手动删除缓存。
- 插件启动后应能自动把旧缓存逐步过渡到新逻辑。

### 8. 自动更新链路保持保存后触发

继续使用现有事件链：

- `vault.on('modify')` -> `indexService.processFileChange(file)`

处理流程保持为：

1. 重新读取该文件最新内容
2. 删除该文件旧的 source blocks 和 references
3. 重新写入该文件新的 source blocks 和 references
4. 对受影响 UUID 重新计算 canonical block
5. 触发 `index-updated`

边界要求：

- 只重算当前文件和受影响 UUID
- 不做全库扫描
- 不做编辑中实时联动

### 9. 渲染层只复用现有刷新机制

保留现有机制，不额外引入更复杂的实时监听：

- Live Preview：继续使用 `indexRevision` 参与 widget signature
- Reading Mode：继续使用 `index-updated` 后的 `refreshOpenMarkdownViews()`

要求：

- 一旦 canonical block 变化，现有引用与嵌入都必须在保存后自动刷新为最新内容。
- 不额外增加逐字符级别的联动刷新。

## 性能要求

1. 单文件保存后，只做单文件解析，不做全库重建。
2. 同 UUID 多源块选择只在内存数组里排序，不能引入跨文件重复读取。
3. 渲染层仍然只消费内存中的 canonical block，不在每次渲染时重新读文件。
4. 不能因为这个修复导致大库中普通打字或滚动明显变卡。

## 验收测试

### 单源块场景

1. 建立标准源块：
   - `- 这是一个块（A）`
   - 下方有 `id:: uuid`
2. 其他位置存在 `((uuid))` 和 `{{embed ((uuid))}}`
3. 修改源块为 `- 这是一个21块（A1）` 并保存
4. 预期：Live Preview 和 Reading Mode 都自动更新为新内容

### 多源块场景

1. 文件 A、文件 B 都有同一个 `id:: uuid`
2. 文件 A 内容旧，文件 B 内容新
3. 修改并保存文件 B
4. 预期：所有引用和嵌入统一显示文件 B 内容

### 多源块反向切换场景

1. 在上一步基础上再修改并保存文件 A
2. 预期：所有引用和嵌入统一切换显示文件 A 内容

### 删除回退场景

1. 删除当前最新源块
2. 若还有其他 active 源块，预期自动回退到剩余源块里最新的那个
3. 若所有 active 源块都没了但引用还在，预期继续进入现有 stale 逻辑

### 外部修改场景

1. 在 Obsidian 外修改源块文件并保存
2. 回到 Obsidian
3. 预期：通过 `modify` 或启动 reconcile 自动同步新内容

### 验证命令

- `npm run build`

### 手动验证范围

- Live Preview
- Reading Mode
- 块引用
- 块嵌入
- Back 菜单
- 源块引用计数徽标

## 明确不做

1. 不做打字过程中的跨页面实时同步
2. 不自动同步旧备份页的原始 Markdown 文本
3. 不自动删除重复 UUID 源块
4. 不改变 `((uuid))`、`{{embed ((uuid))}}`、`id:: uuid` 语法
5. 不放宽源块识别规则
