# 索引与缺失块恢复计划

## 目的

这份文档记录当前插件在“块索引、缺失块回退、缺失块恢复”这一组问题上的产品决策与实现状态，供后续新对话窗口或新的工程代理直接续接。

## 已确认的产品决策

以下决策视为当前固定方案：

1. 缺失源块默认只恢复到专用恢复页。
2. 恢复页路径固定为 `pages/Block Recovery.md`。
3. 默认不尝试恢复回原文件和原行号。
4. 不做删除前拦截或删除前确认。
5. 保护策略依赖于失效回退显示、人工复核工具和恢复页。
6. 解析规则继续保持严格：
   - 只把无序列表块视为候选源块
   - 必须存在缩进属性行 `id:: uuid`
   - 不为了“识别更多情况”而放宽到任意 Markdown 行

## 当前已实现能力

当前代码已经具备：

1. 插件启用后的自动索引。
2. 启动时缓存加载与差异对账。
3. 运行期文件增删改重命名监听。
4. 手动重建索引命令。
5. 状态栏进度与完成状态显示。
6. 反向引用索引。
7. 源块丢失后的 `stale` 回退渲染。
8. 缺失块复核对话框。
9. 恢复到恢复页并保留原 UUID。
10. 大页面 Live Preview 性能优化后的可用版本。

## 当前支持的语法模型

插件服务的是 UUID 风格的大纲式 Markdown 记法。

典型结构：

```md
- 源块内容
  id:: 12345678-1234-1234-1234-123456789abc
```

引用语法：

```md
((12345678-1234-1234-1234-123456789abc))
{{embed ((12345678-1234-1234-1234-123456789abc))}}
```

## 问题本质

当页面出现：

- `[missing block]`
- `Missing block`

本质原因不是宿主应用自身搜索索引失效，而是插件自己的块索引里没有对应 UUID，或者当前缓存与真实库状态不一致。

## 当前设计目标

1. 插件启用后能自动建立可用索引。
2. 插件运行期间能增量更新索引。
3. 用户可手动全量重建索引。
4. 首次索引和手动重建要有清晰进度。
5. 日常增量更新尽量静默。
6. 大库下仍保持可用性能。
7. 源块丢失但仍被引用时，不立即丢失信息。
8. 恢复时保留原 UUID，而不是把引用改写成纯文本。

## 非目标

1. 不依赖宿主应用搜索索引完成块解析。
2. 不自动把所有 `((uuid))` 改写成文本。
3. 不默认把恢复内容插回原始位置。
4. 不做未明确确认的破坏性编辑。

## 索引策略

### 启动阶段

1. 先读持久化缓存。
2. 若无可用缓存，则执行全量重建。
3. 若有缓存，则根据 Markdown 文件元数据做差异对账。
4. 对变更文件增量重解析。
5. 对已删除文件移除活跃源块。
6. 对仍被引用的缺失源块标记为 `stale`。

### 运行阶段

1. 监听 Markdown 文件创建。
2. 监听 Markdown 文件修改。
3. 监听 Markdown 文件删除。
4. 监听 Markdown 文件重命名。
5. 用防抖方式保存缓存。

## 失效块回退策略

当一个曾经存在的源块消失，但引用仍然存在时：

1. 内联块引用显示最后一次缓存到的摘要文本。
2. 块嵌入显示最后一次缓存到的完整内容。
3. 同时给出“源块已丢失，当前显示缓存内容”的提示。

这套设计的目标是防止知识立即断裂。

## 恢复策略

恢复动作只写入恢复页，不尝试猜原位置。

恢复块格式示意：

```md
- 原始块内容
  id:: 原始-uuid
  recovered-from:: 原始文件路径
  recovered-at:: 时间戳
```

如果缓存中有子块内容，一并恢复到该块下方。

## 文档和使用层面需要说明的重点

1. 插件使用自己的块索引，不依赖宿主应用搜索索引。
2. `Rebuild block reference index` 是可见命令，适合在大量外部改动后手动执行。
3. `Review missing source blocks` 用于处理仍被引用但源块丢失的 UUID。
4. 恢复默认进入 `pages/Block Recovery.md`。

## 后续验证重点

1. 大库首次启动体验是否仍可接受。
2. 缓存体积增长是否需要后续优化。
3. 恢复页长期使用后的组织方式是否需要分组。
4. 解析边界是否继续保持足够严格。
