# 右键“剪贴板贴入为大纲”实现记录

## 一、当前结论

旧方案里“自动接管普通粘贴”的路径已经被彻底放弃。根因不是某一条解析规则写错，而是：

- 它直接插进了 Obsidian / CodeMirror 的普通 paste 链路
- 再叠加大纲、属性隐藏、badge 等编辑器扩展后，出现了高 CPU 与界面卡死风险
- 继续在自动 paste 路径上打补丁，不符合稳定性优先原则

现在的正式方案改为：

1. 不拦截普通 `Ctrl/Cmd + V`
2. 不监听 `editor-paste`
3. 只在用户显式右键点击菜单项时执行
4. 只在无序列表块上生效，空列表项也允许显式触发

这让实验功能即使失败，也只影响这一次主动操作，不会拖垮整个编辑器输入链路。

## 二、已实现方案

### 2.1 用户入口

- 设置页新增 `Experimental`
- 新增开关：`Convert pasted content to outline`
- 默认值：`false`
- 只有在该开关开启时，编辑器右键菜单才会出现：
  - `Paste clipboard as outline`

### 2.2 生效条件

右键菜单项只在以下条件全部满足时显示：

- 当前是 Markdown 编辑器
- 当前目标是无序列表项，空列表项也允许
- 不在 fenced code block 内
- 当前文档能正确解析出该列表项的插入位置

这部分后来补了一次稳定性修正：

- 右键目标行不再只依赖 `posAtCoords(x, y)`
- 当前实现会优先按可视行高度定位，再退回坐标命中，最后才用 DOM 行兜底
- 这样空列表项、点在 bullet 附近、点在行尾空白区时，右键菜单更稳定
- 空无序列表项同时放宽为允许：
  - `-`
  - `- `
  - `- 内容`
- 但会继续排除 Markdown 水平分割线 `---`

普通段落、标题、代码块上下文不会显示该菜单项。

### 2.3 剪贴板读取

读取顺序：

1. `text/html`
2. `text/plain`
3. `navigator.clipboard.readText()` 兜底

如果 HTML 可用，就先尝试把它转成 Markdown 再恢复层级；如果 HTML 不可用、过大、结构不足或不可靠，就退回纯文本解析。

### 2.4 转换策略

当前实现采用“优先结构化，失败再降级”的策略：

- HTML 路径：
  - 先检查白名单结构标签
  - 不再只依赖 `htmlToMarkdown()` 后再做列表正则恢复
  - 会优先按受控 DOM 结构提取块序列，再恢复 outline tree
- 纯文本路径：
  - 支持 Markdown 风格无序列表
  - 支持 Markdown 风格有序列表，并统一转成无序列表层级
  - 支持 `#` 标题，保留成 `- # 标题`
  - 支持常见网页复制得到的项目符号字符，例如 `•`
  - 对“正文行 + 后续列表簇 + 再次正文”的结构，会优先恢复成：
    - 正文节点
    - 其子级列表
    - 列表簇结束后的同级正文节点
  - 没有可靠层级时，按非空行平铺成子块

最新一轮修正的重点，不再是继续给单条正则打补丁，而是把层级恢复改成更稳定的“段落块 + 列表簇”模式。这样像下面这种网页 / ChatGPT 内容：

```md
不需要吹，不需要推广腔。最有力的是：
  - “Logseq MD 用户迁 Obsidian：块引用和块嵌入如何保留？”
  - 讲清楚：
    - 什么能保留；
这样它才会从“一个插件”变成“一个迁移方案”。
```

会优先恢复成：

```md
- 不需要吹，不需要推广腔。最有力的是：
	- “Logseq MD 用户迁 Obsidian：块引用和块嵌入如何保留？”
	- 讲清楚：
		- 什么能保留；
- 这样它才会从“一个插件”变成“一个迁移方案”。
```

### 2.5 插入策略

插入目标固定是“当前右键目标无序列表项的子块区域”：

- 如果当前块已有直接子块：
  - 新贴入内容插在第一个直接子块之前
- 如果当前块没有直接子块：
  - 新贴入内容插在当前块 continuation 尾部之后

不会：

- 删除当前目标块
- 覆盖当前目标块正文
- 跨越并破坏现有子树

### 2.6 性能保护

普通内容直接处理，限制为：

- HTML 最大 `128 KiB`
- 纯文本最大 `512 KiB`
- 最大输入行数 `5000`
- 最大输出块数 `3000`
- 最大层级深度 `32`
- 最大生成 Markdown `512 KiB`
- 总处理超时 `10s`

处理过程分阶段显示 Notice：

- `Reading clipboard...`
- `Converting clipboard to outline...`
- `Inserting outline...`

任何限制命中时都直接中止，不写入半成品。

较大内容不再直接拒绝，而是先显示只有两个选项的确认框：

- `Process`
- `Cancel`

用户确认后使用较大但仍受控的处理档位：

- HTML 最大 `2 MiB`
- 纯文本最大 `2 MiB`
- 最大输入行数 `20000`
- 最大受支持 HTML 标签数 `50000`
- 最大输出块数 `10000`
- 最大层级深度 `64`
- 最大生成 Markdown `1 MiB`
- 总处理超时 `60s`

HTML 标签数与纯文本行数分别计数，不再错误地用文本行数上限限制 HTML 标签。DOM 可用时会优先按结构分片解析，不会在开始时同步把整份大 HTML 先交给 `htmlToMarkdown()`。

处理过程中按约 `4ms` 时间片主动让出主线程，并复用同一条右上角 Notice 更新百分比。任何取消、超时或硬限制命中都不会写入半成品。

### 2.7 原位置锚定与跨页面处理

右键动作触发时，在读取剪贴板之前就记录：

- 原始 `TFile`
- 目标列表行原文
- 目标祖先列表路径
- 前后非空上下文
- 原文档摘要与初始行号

转换完成后不会使用旧字符偏移直接写入，而是重新读取原文件并定位：

1. 原文档未变化时使用原目标。
2. 文件其他位置发生变化时，按目标行、祖先路径与前后上下文重新寻找唯一位置。
3. 目标删除、出现同分候选或无法唯一定位时安全取消。

原文件仍在编辑器中时只执行一次 `replaceRange`；原文件已关闭时只执行一次 `vault.process()`。用户可在处理期间切换页面，插件不会强制切回，也不会把结果写入当前新页面。

同一时间只允许一个大纲贴入任务。插件被关闭时会取消正在运行的任务并清理状态提示。

## 三、当前边界

### 3.1 已明确支持

- ChatGPT “复制按钮”得到的规范 Markdown
- 网页复制得到的 HTML 列表，在结构足够明确时恢复层级
- 普通多行文本转成多个子块
- 标题转成 `- # 标题`
- 有序列表层级转成无序列表层级

### 3.2 暂不承诺完全还原

- 任意复杂网页富文本的全部细节
- 所有 HTML 表格、引用、代码块都百分之百按原网页视觉还原
- 非结构化网页框选文本的“猜层级”

如果剪贴板本身没有可靠层级信息，当前实现会降级，不会冒险推断。

## 四、验证要求

当前仓库应至少通过：

```powershell
npm test
npm run build
git diff --check
```

关键手动验证：

1. 在无序列表块上右键执行 `Paste clipboard as outline`
2. 在空无序列表项 `-` 或 `- ` 上右键，确认菜单也能出现
2. 粘贴标准 Markdown 多级列表，正确生成子块层级
3. 粘贴 ChatGPT 复制按钮得到的内容，层级基本正确
4. 粘贴网页框选内容，重点验证“正文行 + 列表簇 + 尾部正文”不会被压平
5. `Copy current level and children` 在空列表项和普通列表项上都能稳定出现
6. 关闭设置开关后，普通粘贴完全不受影响

## 五、后续演进方向

如果后续要继续增强，只能沿着下面的方向做：

1. 保持“显式右键动作”这个入口，不回到自动 paste 拦截
2. 继续扩大 HTML 白名单结构支持
3. 继续补真实网页样本测试
4. 如需更重的 HTML 处理，再考虑迁移到 Worker，但不能影响普通编辑链路

禁止回退到旧方案：

- `editor-paste`
- DOM `paste` 监听
- 交易式自动改写普通粘贴
- 任何会让用户一按 `Ctrl/Cmd + V` 就直接进入大计算的路径

## 八、1.3.17 合并发布范围

远程上一正式版本为 `1.3.15`。本次以 `1.3.17` 一次发布两轮连续改进，不单独发布中间版本：

1. 输出规范化：删除普通空白分隔行，保证无序列表连续；所有生成的子级列表严格按“一层一个 `Tab` + `- `”输出，同时保留 fenced code block 内必要空行与相对缩进。
2. 大内容安全处理：内容较大时提供 `Process / Cancel` 确认、分时处理和进度提示；处理前记录原文件与大纲位置，切换页面后仍写回原位置，目标失效或定位歧义时安全取消。

最终发布产物和文档统一使用版本号 `1.3.17`。

## 六、本轮增强补充

这轮继续沿用上面的“显式右键触发”总原则，但补充了三条更稳定的细化规则：

1. 新生成的子级无序列表统一使用 `Tab` 缩进，不再默认回退成两个空格。
2. HTML 路径不再只依赖 `htmlToMarkdown()` 后再做正则恢复，而是优先从受控 DOM 结构直接建树：
   - `ul / ol / li`
   - `p`
   - `h1-h6`
   - `blockquote`
   - `pre / code`
   - `table`
3. 新增右键菜单项：
   - `Copy current level and children`

`Copy current level and children` 的边界固定为：

- 直接从当前编辑器文本提取子树，不依赖块索引缓存
- 会保留：
  - 当前列表行
  - 软换行内容
  - 属性键值行
  - continuation 下的 fenced code block
  - 所有子级无序列表及其子树
- 输出时只去掉被复制根节点自身的外层缩进
- 不导出渲染后的文本，而是导出原始 Markdown

## 七、空行清理与严格 Tab 层级

为了避免网页和 ChatGPT 复制内容中的排版空行打断 Markdown 列表，本功能增加以下输出不变量：

1. 普通空行以及只包含空格或 `Tab` 的行不会生成节点，也不会作为 continuation 输出。
2. HTML、纯文本和降级解析完成后，都必须经过同一层树结构清理：
   - 删除空白节点
   - 删除普通空白 continuation
   - 空白父节点如果只有子节点，则提升子节点，不制造空的 `- `
3. 剪贴板中的 2 空格、4 空格、`Tab` 或混合缩进只负责判断原始父子关系，不直接复用到最终 Markdown。
4. 最终列表缩进只由树深度决定：
   - 当前插入层使用目标块的子级前缀
   - 每增加一级子节点，严格增加一个 `Tab`
   - 每个列表节点统一使用 `- ` 标记
5. 普通 continuation 会去掉剪贴板携带的公共结构缩进，防止插入后重复缩进。
6. fenced code block 内部空行作为唯一例外保留，代码内容的相对缩进也保留；该例外不会改变外围无序列表的连续层级。

必须持续覆盖的回归样本：

- 每个列表项之间都有单个空行
- 连续多个空行或只含空格、`Tab` 的空行
- 2 空格、4 空格、`Tab` 混合的多级列表
- 代码围栏内包含空行和相对缩进
- 空白 HTML 容器包裹真实子列表
