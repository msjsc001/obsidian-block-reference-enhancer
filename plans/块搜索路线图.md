# 块搜索路线图

## 文档目的

这份文档记录“插件独立块搜索功能”的当前决策、实现方向、阶段计划和复杂度评估。

即使当前聊天窗口丢失，新的维护者或编程助手也应能根据本文继续推进该功能。

## 当前状态

- 优先级：中低
- 当前阶段：路线图与方案设计
- 当前结论：暂不进入立即开发排期
- 推荐实现方式：插件自建独立搜索框或搜索视图
- 不推荐实现方式：接入或修改 Obsidian 原生 Search 核心插件内部逻辑

## 背景问题

插件当前能把 UUID 风格语法渲染成真实块内容：

```md
((uuid))
{{embed ((uuid))}}
```

但是 Obsidian 原生搜索搜索的是文件里的原始 Markdown 内容。文件里实际存在的是 UUID 语法，而不是渲染后的源块文字。

因此，当用户搜索“源块真实文字”时：

1. Obsidian 原生搜索通常只能搜到源块本身。
2. Obsidian 原生搜索无法稳定把引用处的 `((uuid))` 当成源块文字来搜索。
3. 块嵌入和块引用虽然视觉上显示了源块内容，但这个渲染结果不属于原始文件文本。

## 已确认的产品决策

1. 第一阶段不接 Obsidian 原生 Search 的内部实现。
2. 第一阶段做插件自己的独立搜索入口。
3. 搜索目标是“真实块内容”，不是原始 UUID 字符串。
4. 搜索依赖插件已经维护的本地块索引，不重新全库扫描一套。
5. 搜索结果必须能跳转到源块或引用位置。
6. 大库性能优先，默认限制结果数量，并使用输入防抖。
7. 后续可以提供“把 UUID 查询发送到 Obsidian 原生搜索”的辅助能力，但不作为第一版核心。

## 为什么不直接接入 Obsidian 原生搜索

理想情况下，Obsidian 如果提供类似以下稳定 API，会是最佳方案：

```ts
search.registerProvider(...)
search.registerIndex(...)
```

但目前公开插件 API 中没有稳定的“注册自定义搜索源 / 自定义搜索索引 / 自定义搜索结果提供者”的入口。

如果强行接入原生搜索，大概率要依赖以下不稳定方式：

1. 访问核心 Search 插件内部对象
2. 修改或拦截 Search 查询流程
3. 读取或改写 Search 结果 DOM
4. 依赖内部类名、内部状态或非公开方法

这些做法短期可能能做出效果，但维护风险高。Obsidian 或核心 Search 插件更新后，内部实现变化可能直接导致功能失效。

## 推荐方案

推荐做一个插件自己的块搜索系统。

用户体验上，它可以像一个轻量 Quick Switcher 或搜索弹窗：

1. 用户通过命令面板打开 `Search UUID blocks`
2. 输入源块真实文字
3. 插件搜索自己的本地块索引
4. 结果显示源块摘要、文件路径、行号、UUID、引用次数
5. 点击结果跳转到源块
6. 结果中可进一步查看引用位置或嵌入位置

这个方案的优点：

1. 不碰 Obsidian 原生 Search 内部实现
2. 维护风险低
3. 性能可控
4. 能充分利用现有索引
5. 结果展示可以更适合 UUID 块引用、块嵌入场景

## 第一版目标

第一版只解决最核心的问题：

用户记得源块真实内容，但不知道源块在哪里。

### 第一版入口

新增命令：

```text
Search UUID blocks
```

命令行为：

1. 打开插件自己的搜索弹窗或搜索视图
2. 输入框自动聚焦
3. 用户输入关键词后搜索已索引源块
4. 点击结果跳转到源块所在文件和行号

### 第一版搜索范围

只搜索：

1. 已建立源块的正文内容
2. 已建立源块的文件路径
3. 已建立源块的 UUID

第一版不搜索：

1. 普通未建立源块的 Markdown 文本
2. PDF 内容
3. 图片 OCR 内容
4. Obsidian 原生搜索索引
5. 未被插件识别为源块的松散 Markdown 行

这个限制是刻意设计的。它能避免大库里每次搜索都重新扫全库，也符合插件当前“只处理标准 UUID 源块”的定位。

## 第二版目标

第二版再补充引用关系搜索。

新增命令可以是：

```text
Search UUID block references
```

或者在同一个搜索界面里提供范围筛选：

```text
All / Source blocks / References / Embeds
```

第二版支持：

1. 搜索源块真实内容
2. 展示哪些页面引用了该源块
3. 展示哪些页面嵌入了该源块
4. 点击引用结果跳转到引用处
5. 点击嵌入结果跳转到嵌入处

## 数据来源

搜索不重新构建一套独立索引，而是复用当前插件的本地索引。

现有索引已经能提供：

1. UUID 对应的源块正文
2. 源块所在文件路径
3. 源块起始行号
4. 同一 UUID 的多个活动源块位置
5. 每个 UUID 的块引用位置
6. 每个 UUID 的块嵌入位置
7. 引用次数
8. 源块是否 stale 或 missing

如果现有索引缺少某些字段，应优先在索引服务中补只读查询方法，而不是单独扫描文件。

## 搜索文档模型

内部可以把可搜索内容抽象成统一条目：

```ts
type BlockSearchEntry = {
  kind: "source" | "reference" | "embed";
  uuid: string;
  searchableText: string;
  previewText: string;
  filePath: string;
  line: number;
  referenceCount: number;
  stale: boolean;
};
```

第一版可以只生成 `kind: "source"`。

第二版再生成 `reference` 和 `embed`。

## 搜索匹配策略

第一版保持轻量：

1. 查询输入做 trim
2. 空查询不搜索或显示最近源块
3. 默认忽略大小写
4. 使用简单包含匹配作为基础
5. 可选使用 Obsidian 公开的 `prepareSimpleSearch`
6. 大库中谨慎使用模糊搜索

后续可加入：

1. 命中词高亮
2. 多关键词 AND 匹配
3. 路径匹配加权
4. UUID 精确匹配
5. 引用次数加权
6. 最近修改加权

## 排序规则

第一版排序建议：

1. 源块正文完全包含查询词的结果优先
2. 源块正文开头命中的结果优先
3. 文件名或路径命中的结果次之
4. 引用次数更多的结果略微靠前
5. 文件路径和行号作为最终稳定排序

不要第一版就做复杂相关性评分。先保证结果可解释、稳定、不卡顿。

## 界面设计

第一版建议使用 Modal，而不是右侧固定 View。

原因：

1. 实现更小
2. 更接近命令式搜索
3. 不需要处理复杂侧边栏状态
4. 更容易在桌面和移动端保持可用

### Modal 第一版布局

1. 顶部输入框
2. 中间结果列表
3. 每条结果显示：
   - 源块摘要
   - 文件名或路径
   - 行号
   - 引用次数
   - UUID 短码
4. 底部显示结果数量和限制提示

### 交互要求

1. 打开后输入框自动聚焦
2. 输入防抖 150-250ms
3. 上下方向键选择结果
4. Enter 跳转到选中结果
5. Esc 关闭
6. 鼠标点击结果跳转

## 跳转行为

源块结果点击后：

1. 打开对应 Markdown 文件
2. 定位到源块起始行
3. 滚动到可见位置

如果同一个 UUID 有多个活动源块：

1. 第一版可以直接显示多条源块结果
2. 每条结果对应一个具体文件和行号
3. 不自动猜测哪个是主源块

如果源块已缺失：

1. 默认不显示在普通结果里
2. 后续可以通过筛选显示 stale/missing 结果
3. 不在搜索动作里自动恢复源块

## 性能规则

1. 搜索只读内存索引，不触发全库扫描
2. 输入必须防抖
3. 每次最多渲染前 50 条结果
4. 大结果集使用分页或虚拟列表
5. 搜索时不要读取文件内容
6. 搜索时不要重新解析 Markdown
7. 搜索条目缓存可以在索引更新后统一失效
8. 不在每次按键时重建大型数据结构

## 与 Obsidian 原生搜索的关系

第一版不接入原生搜索。

后续可以增加一个低风险辅助能力：

1. 用户在插件搜索里搜真实块内容
2. 插件找到对应 UUID
3. 用户点击“Search references in Obsidian”
4. 插件打开或填入 Obsidian 原生搜索查询：

```text
((uuid)) OR {{embed ((uuid))}}
```

这不是让 Obsidian 原生搜索理解源块文字，而是插件先把真实文本翻译成 UUID，再借用原生搜索查原始语法。

这类辅助能力可以考虑，但不是第一版必需项。

## 实施阶段

### 阶段 1：最小可用搜索

目标：

1. 新增 `Search UUID blocks` 命令
2. 新增搜索 Modal
3. 复用现有索引搜索源块正文
4. 显示前 50 条结果
5. 点击或 Enter 跳转源块

建议验证：

1. 小库中搜索源块正文能命中
2. 大库中输入不卡顿
3. 点击结果能跳转到准确行
4. 多源块同 UUID 时显示多个源位置
5. `npm run build` 通过

### 阶段 2：结果质量与引用关系

目标：

1. 增加命中词高亮
2. 增加引用次数显示
3. 增加源块 / 引用 / 嵌入筛选
4. 增加引用位置结果
5. 增加嵌入位置结果

建议验证：

1. 引用结果跳转到引用处
2. 嵌入结果跳转到嵌入处
3. 大量引用时列表分页或限制仍然流畅
4. 结果排序符合直觉

### 阶段 3：高级体验

目标：

1. 右侧固定搜索 View
2. 搜索历史
3. 路径筛选
4. stale/missing 结果筛选
5. 复制结果
6. 把 UUID 查询发送到 Obsidian 原生搜索

阶段 3 不建议在第一轮一起做。它们会显著增加 UI 状态和维护成本。

## 与“返回源块 Back 按钮”的复杂度对比

搜索功能整体复杂度会高于“鼠标悬浮 Back 返回源块”。

Back 功能的复杂点主要在：

1. Live Preview 和 Reading Mode 都要显示
2. 悬浮、点击保持、焦点状态要稳定
3. 单源块、多源块、源块缺失要处理
4. 点击时精确跳转

但 Back 功能的数据路径相对简单：

```text
当前 UUID -> 查源块位置 -> 跳转
```

搜索功能的数据路径更长：

```text
用户输入 -> 查询解析 -> 搜索条目生成或缓存 -> 匹配 -> 排序 -> 截断 -> 渲染结果 -> 键盘交互 -> 跳转
```

搜索还会多出这些额外问题：

1. 大库性能
2. 搜索排序是否符合用户预期
3. 结果摘要如何截取
4. 命中词如何高亮
5. 结果过多时如何分页或虚拟列表
6. 搜索源块还是引用位置的产品边界
7. Modal 和未来固定 View 的状态管理

因此，搜索功能不是代码量最大的问题，而是产品边界和性能边界更复杂。

## 工期评估

以下估算以当前插件已有索引、跳转能力、引用位置数据为基础。

### 最小可用版

范围：

1. 命令入口
2. 搜索 Modal
3. 搜源块正文
4. 显示基础结果
5. 点击跳源块

预计时间：

- 0.5 到 1 天可完成基本实现
- 1 到 2 天可完成较稳妥版本、构建验证和基础文档

复杂度相对 Back 功能：

- 约为 Back 功能的 1.2 到 1.5 倍

### 实用完整版

范围：

1. 源块搜索
2. 引用位置搜索
3. 嵌入位置搜索
4. 结果筛选
5. 高亮摘要
6. 引用次数显示
7. 大库结果限制和分页
8. 文档同步

预计时间：

- 3 到 5 天较稳妥

复杂度相对 Back 功能：

- 约为 Back 功能的 2 到 3 倍

### 高级版

范围：

1. 固定侧边栏搜索 View
2. 搜索历史

## 推荐下一步

未来真正开发时，建议先做阶段 1。

不要第一轮就做完整搜索系统。阶段 1 只要把“搜源块真实内容并跳转”做好，就已经能解决当前最核心的缺口。

阶段 1 稳定后，再根据实际使用反馈决定是否进入阶段 2。

## 非目标

第一轮明确不做：

1. 不修改 Obsidian 原生 Search 插件
2. 不访问原生 Search 内部对象
3. 不改写用户 Markdown 文件来制造可搜索文本
4. 不生成影子索引 Markdown 文件
5. 不搜索整个 vault 的普通文本
6. 不引入大型全文搜索依赖
7. 不做复杂语义搜索
8. 不把搜索结果自动恢复或改写源块

## 风险点

1. 大库搜索如果每次都重建条目，可能造成输入卡顿。
2. 搜索结果排序如果过于简单，用户可能觉得“不够聪明”。
3. 引用和嵌入结果如果第一版一起做，范围容易扩大。
4. 如果后续强行接原生 Search 内部实现，维护风险会明显上升。
5. 如果搜索普通 Markdown 文本，会偏离插件定位并增加扫描成本。

## 验证清单

未来实现时至少验证：

1. `npm run build` 通过
2. 小库搜索源块正文命中正确
3. 大库连续输入无明显卡顿
4. 点击结果能跳到源块准确行
5. 多个文件存在同 UUID 源块时，结果不会丢失
6. 源块删除后搜索结果不会误跳
7. 插件关闭后不会留下事件监听或 DOM 残留
8. 中英文 README 路线图同步更新
